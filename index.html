Inside Job — leaking the admin's note through a CSP report oracle

No script ever runs on the target. A strict nonce-based CSP is bypassed by chaining a reflected HTML-injection sink, dangling-markup capture, an attacker-controlled report-uri, and a credentialed cross-origin read.

$ cat ./00-intro.md

A notes app you can't XSS the normal way

The June 2026 Intigriti challenge, “Inside Job”, is a private-notes application. The objective is to read the admin's private note — it holds the flag. The complication: every response ships a strict, per-request nonce-based CSP, so the usual <script> / alert() route is dead on arrival.

The solution is scriptless: not one line of JavaScript executes on the challenge origin. It runs entirely on injected HTML, the browser's own CSP violation reporting, and a credentialed cross-origin read — three small flaws that compose into a full confidentiality breach with zero clicks.

i

The admin's note contains the reminder “Check CSP reports daily.” That's the author pointing at the intended channel: the CSP report endpoint is the exfiltration path.

$ cat ./01-recon.md

Read the response before touching a payload

The endpoint of interest is GET /search?owner=<o>&q=<q>&description=<d>. The description parameter is reflected unencoded into the head:

reflection · /search<meta name="description" content="Notes search — [description]">

And the CSP header:

header · content-security-policydefault-src 'none';
script-src  'nonce-RANDOM';
style-src   'nonce-RANDOM';
form-action 'self';  base-uri 'none';
report-uri  /csp-report/[owner]

Two facts decide everything. The nonce is random per response, so injected scripts can't run. And the report-uri is built from owner — the attacker chooses where violation reports are stored.

One rendering quirk seals it. When q matches no note, the app renders all notes (content HTML-escaped — never a literal '), then reflects q raw:

body · no match<article class="note"><strong>title</strong><p>content</p></article>
...
<p>[q] not found</p>

Unencoded reflection at the top, raw reflection at the bottom, and no literal quote in between — the exact shape a dangling-markup capture needs.

$ cat ./02-insight.md

Why a strict CSP doesn't save you

Under default-src 'none'; script-src 'nonce-…' it feels like nothing can leave: no inline script, no external load, no fetch to an attacker domain. As an outbound channel from the challenge origin, that holds.

But CSP has a side effect: when it blocks a sub-resource it emits a report containing the blocked-uri — the URL the page tried to load. Force a secret into a URL the browser attempts to fetch, and CSP blocks the fetch while faithfully recording the secret-bearing URL.

The reporting feature becomes a data oracle. We don't exfiltrate from the challenge origin at all — we get the secret into a blocked-uri, then read the report through a second channel. That second channel is the CORS bug.

Three flaws — each minor, chained fatal

01Reflected HTML injection (scriptless XSS) on /search

confirm/search?owner=x&q=test&description="><b>INJECTED</b>

<b>INJECTED</b> lands raw in <head>. The nonce blocks execution, so it's a scriptless XSS — no JS, arbitrary markup. Two primitives: a CSP-allowed <meta http-equiv="refresh"> (bounce the bot) and an unclosed <img> whose src swallows markup (capture notes).

02Attacker-controlled CSP report-uri

Since report-uri /csp-report/[owner] is built from owner, setting owner=admin routes a violation generated on the admin's page into /csp-report/admin — a bucket we read later.

Dangling-markup capture, step by step

The goal: every note on the admin's page — flag included — inside one URL the browser will try to load. The primitive is dangling markup: open an attribute value and never close it, so the parser greedily eats following markup as part of it.

We use the two reflections as a pair:

RoleParamValue
Open the unclosed srcdescription"><img src='/LK
Close it after all notesq'
 
rendered · admin iframe<meta name="description" content="Notes search — "><img src='/LK">
  ...
  <article class="note"><strong>INTIGRITI{019ea42e-...}</strong></article>
  ...

The single-quoted src opens at /LK. Note content is escaped with no literal ', so the attribute runs all the way to the ' supplied via q in the “not found” line — swallowing every note into the URL. default-src 'none' blocks the image and fires a report whose blocked-uri now carries the notes.

!

In blocked-uri the flag is URL-encoded — { → %7B, } → %7D. A naïve INTIGRITI\{…\} regex misses it; decode the report first.

$ cat ./09-remediation.md

Break any one link, defeat the chain

  • Encode the reflected input (root cause)

    HTML-attribute-encode " ' < > & in description before it reaches the <meta> attribute, and likewise for the raw q reflection.

  • Don't derive report-uri from user input

    Use a fixed, server-controlled report endpoint and bind reports to the authenticated session server-side — never to a path segment the requester chooses.

  • Fix the CORS policy

    Stop reflecting Origin with Allow-Credentials: true. Make /csp-report/* same-origin and owner-only, or restrict to a trusted allow-list. Never pair a reflected origin with credentials.

  • Harden the session cookie

    Set SameSite=Lax (or Strict) unless cross-site auth is required. With Lax, the credentialed cross-origin read no longer carries the admin's session.

$ cat ./10-conclusion.md

Controls compose — so do their failures

A strict, nonce-based CSP stops script execution, but not markup injection — and its own violation reports become a data oracle when a secret can be coerced into a blocked-uri. Pair that with dangling-markup capture (strongest precisely when JS is off the table) and a credentialed CORS read, and a set of individually “minor” findings becomes a zero-click, cross-browser confidentiality breach.

Reviewing each control in isolation misses exactly this class of bug. The impact lives in the chain.
